# ТЗ: Постоянное хранилище коммитов + инкрементальные отчёты по репозиториям

## 0. Контекст (что есть сейчас)

- `collect_commits()` (`src/core/collector.py`) при каждом `/analyze` **пересобирает весь период заново** через
  GraphQL/REST.
- `src/github/cache.py` — repo-level кеш в Redis с TTL (`cache_ttl_github`), инвалидируется по `pushed_at`. Это кеш *
  *ускорения одного запроса**, не хранилище истории.
- `commit_cache` таблица в `src/storage/database.py` уже объявлена (`get_cached_commits`/`set_cached_commits`), но *
  *нигде не вызывается** — мёртвый код, задел под это ТЗ.
- Дедупликация запросов (`find_existing_requests` + `get_github_fingerprint`) работает на уровне "весь профиль
  изменился/не изменился", а не на уровне отдельных коммитов.
- Бот не умеет анализировать один конкретный репозиторий — только весь профиль пользователя.

## 1. Цель

1. Коммиты сохраняются в БД **навсегда** (не TTL), с нормализованной схемой, позволяющей быстро фильтровать по
   репозиторию.
2. Повторный отчёт по уже известному периоду формируется **из БД**, без похода в GitHub API (цель — десятки/сотни мс).
3. Данные обновляются **инкрементально**: либо push-вебхуком (реалтайм), либо периодическим poll-диффом (universal
   fallback) — но никогда полным пересбором истории с нуля.
4. Отчёт можно получить (а) по всем репозиториям сразу, (б) по одному выбранному репозиторию.

## 2. Важное ограничение (решает архитектуру)

Бот сейчас читает данные **своим** токеном (`GITHUB_TOKEN`/`GITHUB_EXTRA_TOKENS`), а не токеном анализируемого
пользователя. Установка GitHub-вебхука требует `admin:repo_hook`-права на конкретный репозиторий — которых у бота нет и
не будет для произвольного `@username`.

Отсюда — **два независимых уровня**, не путать:

| Уровень                          | Кому подходит                                                                                                    | Что даёт                                                                                                         |
|----------------------------------|------------------------------------------------------------------------------------------------------------------|------------------------------------------------------------------------------------------------------------------|
| **A. Инкрементальный poll-diff** | Любой публичный профиль, без доп. прав                                                                           | Не бьёт полный период заново, только "с момента последней синхронизации". Уже даёт основной выигрыш по скорости. |
| **B. Push-вебхук**               | Только если пользователь сам подключил свой аккаунт (OAuth) и дал права на hook, ИЛИ это репозитории самого бота | Реальный realtime, отчёт готов ещё до нажатия кнопки                                                             |

**Рекомендация: делать A как MVP (этап 1), B — опциональная надстройка (этап 2) для тех, кто явно захочет реалтайм.**
Ниже расписаны оба, но приоритет — этап 1.

## 3. Модель данных

Новые таблицы в `src/storage/database.py` (SQLAlchemy Core, тот же паттерн, что и `requests`/`user_bindings`):

### `tracked_repos`

| поле                | тип                       | описание                                        |
|---------------------|---------------------------|-------------------------------------------------|
| `repo_full_name`    | Text, PK                  | `owner/name`                                    |
| `owner_login`       | Text                      | владелец репо на GitHub                         |
| `analyzed_username` | Text, часть составного PK | чей профиль отслеживаем (author filter)         |
| `default_branch`    | Text                      |                                                 |
| `last_synced_sha`   | Text nullable             | последний обработанный коммит на default_branch |
| `last_synced_at`    | Text (ISO)                | когда последний раз синхронизировали            |
| `last_pushed_at`    | Text (ISO)                | `pushedAt` репозитория на момент синхронизации  |
| `sync_mode`         | Text                      | `poll` \| `webhook`                             |
| `webhook_id`        | Text nullable             | id хука на GitHub, если `sync_mode=webhook`     |
| `is_active`         | Boolean                   | отслеживание включено/выключено                 |
| `needs_full_resync` | Boolean, default False    | флаг после force-push/рассинхрона               |
| `created_at`        | Text                      |                                                 |

PK: (`repo_full_name`, `analyzed_username`) — один и тот же репо теоретически может отслеживаться в контексте разных
анализируемых юзеров (участник + автор).

### `commits` (замена мёртвой `commit_cache`, нормализованная, не JSON-blob)

| поле                | тип                                                                                      |
|---------------------|------------------------------------------------------------------------------------------|
| `sha`               | Text                                                                                     |
| `repo_full_name`    | Text                                                                                     |
| `analyzed_username` | Text                                                                                     |
| `committed_at`      | Text (ISO)                                                                               |
| `message`           | Text (subject, как сейчас — 72 симв. хватит на этапе сериализации, в БД можно чуть шире) |
| `additions`         | Integer                                                                                  |
| `deletions`         | Integer                                                                                  |
| `changed_files`     | Integer                                                                                  |
| `files_json`        | Text nullable                                                                            | компактный список файлов (`[[stem, ext, +, -], ...]`), тот же формат, что уже строит `models.py::_file_row` — переиспользуем |
| `created_at`        | Text                                                                                     | когда запись появилась в БД |

PK: (`sha`, `repo_full_name`, `analyzed_username`).
Индексы: `(repo_full_name, analyzed_username, committed_at)`, `(analyzed_username, committed_at)`.

Почему `files_json` inline, а не отдельная таблица `commit_files`: отчёт и так собирается по формату `CompactResult`,
где файлы уже финальный компактный массив. Нормализация файлов в отдельную таблицу нужна только если в будущем
понадобится фильтр "все коммиты, тронувшие файл X" — сейчас такого требования нет, не усложняем.

Удалить/задепрекейтить: `commit_cache` (старая нереализованная таблица) — заменяется таблицей `commits` выше.
`src/github/cache.py` (Redis TTL-кеш) можно оставить как есть или выпилить — с постоянным хранилищем в БД он теряет
смысл.

## 4. Этап 1 — инкрементальный poll-diff (MVP)

### 4.1 Логика синхронизации репозитория

Новая функция, условно `src/core/sync.py::sync_repo_commits(service, repo, analyzed_username)`:

1. Прочитать `tracked_repos` для `(repo.full_name, analyzed_username)`.
2. Если записи нет **или** `needs_full_resync=True` → полный backfill (текущий `collect_commits`-путь для одного репо)
   на весь запрошенный период, записать в `commits`, создать/обновить `tracked_repos`.
3. Если запись есть и `repo.pushed_at <= tracked.last_pushed_at` → ничего не делать, данные уже актуальны.
4. Если `repo.pushed_at > tracked.last_pushed_at` → **докачать только новые коммиты**:
   `GitHubService.get_commit_history(repo, since=tracked.last_synced_at)`, отфильтровать уже существующие `sha` (upsert
   идемпотентен — можно просто upsert-ить без предварительной проверки), обогатить файлами через `enrich_with_details`
   как сейчас, записать в `commits`, обновить `last_synced_sha`/`last_synced_at`/`last_pushed_at`.

Это фактически замена `get_github_fingerprint`-подхода: вместо "весь профиль совпадает по хэшу — берём старый JSON
целиком" → "по каждому репозиторию отдельно проверяем pushed_at и докачиваем дельту". Более гранулярно, не зависит от
объёма остального профиля.

### 4.2 Изменение пайплайна `analyze_github_user` (`src/worker/tasks.py`)

Заменить блок «дедупликация по fingerprint всего профиля» на:

1. `list_repositories(username)` → список активных репо за период.
2. Для каждого репо — `sync_repo_commits(...)` (можно с тем же `asyncio.Semaphore(max_concurrency)`, что и сейчас в
   `collector.py`).
3. Собрать `AnalysisResult` не через `collect_commits`, а через SQL-выборку из таблицы `commits` за нужный период (см.
   4.4) — то есть сборка и синхронизация теперь разделены: синхронизация обновляет БД, сборка отчёта — чистый read из
   БД.

### 4.3 Кнопка "быстрый отчёт" (0.1 сек)

Отдельный путь без постановки arq-задачи вообще, для случая "данные точно свежие" (например, юзер только что смотрел
отчёт минуту назад):

- Проверка "свежести" — сравнить `tracked_repos.last_synced_at` со временем последнего вебхука/синка; если моложе N
  минут (настраиваемый порог, например `cache_ttl_report`) — отдать отчёт прямо из БД синхронно в хендлере, без
  ARQ-очереди.
- Если "протухло" — как обычно ставим job в очередь (он сходит в GitHub за дельтой и обновит БД).

Это и даёт ощущение "мгновенного" отчёта, не требуя вебхуков: чем чаще юзер спрашивает, тем чаще БД тёплая.

### 4.4 Генерация отчёта из БД (per-repo / all)

Новая функция
`src/core/report.py::build_analysis_result(username, period_start, period_end, repo: str | None = None) -> AnalysisResult`:

```
SELECT * FROM commits
WHERE analyzed_username = :username
  AND committed_at BETWEEN :period_start AND :period_end
  AND (:repo IS NULL OR repo_full_name = :repo)
ORDER BY committed_at
```

Дальше — те же `to_compact()`/`serialize_result()` из `src/models/models.py`, без изменений формата (LLM-промпт, размер
отчёта — всё как есть).

## 5. Этап 2 — вебхуки (опционально, для тех кто явно подключился)

Актуально только если добавляете **явное подключение своего GitHub-аккаунта** (OAuth, отдельная фича, не относится к "
анализ произвольного @username чужим токеном").

### 5.1 Компонент приёма

- Отдельный сервис `src/webhook/` (aiohttp, свой Dockerfile-target/CMD, свой сервис в `docker-compose.yml`), публично
  доступен через reverse-proxy.
- `POST /webhook/github/{repo_full_name}` — валидация `X-Hub-Signature-256` по секрету из
  `tracked_repos.webhook_secret` (или общий `WEBHOOK_SECRET` из `.env` для MVP).
- Хендлер **не обрабатывает payload синхронно** — сразу отвечает 200 и кидает событие в Redis (
  `publish("webhook:push", payload)` или отдельная arq job `handle_push_event`), т.к. GitHub требует ответ <10с, а
  обогащение файлами через REST может занять больше.

### 5.2 Обработка `push`-события

- Фильтр: только `ref == refs/heads/{default_branch}`.
- Важный нюанс формата GitHub payload: `commits[]` содержит `added/removed/modified` (имена файлов), **но не строки +/-
  **. Для additions/deletions по-прежнему нужен точечный REST-запрос `enrich_with_details` на каждый новый sha — но их
  обычно 1–5 за push, дёшево.
- `forced: true` → выставить `tracked_repos.needs_full_resync = True`, не пытаться угадать дельту, при следующей
  синхронизации — полный ресинк этого репо.
- Если `commits[]` пуст, а `head_commit` есть (GitHub схлопывает список при >20 коммитах в одном push) → зафетчить
  `GET /repos/{owner}/{repo}/compare/{before}...{after}`.

### 5.3 Регистрация вебхука

`POST /repos/{owner}/{repo}/hooks` от лица токена **владельца репозитория** (не токена бота) — то есть только в
сценарии, где юзер сам авторизовался. Требует отдельный OAuth-флоу и хранение пользовательского access-токена — это по
объёму отдельное ТЗ, тут просто фиксируем как предпосылку.

### 5.4 Safety net

Вебхуки теряются (даунтайм сервиса, сетевые сбои). Нужна периодическая arq cron-задача (`WorkerSettings.cron_jobs`, в
arq это поддерживается) — раз в N часов сверять `pushed_at` каждого `tracked_repos` с `last_pushed_at`, и если
разошлось — тихо доехать через тот же `sync_repo_commits` из этапа 1. Т.е. этап 2 **не заменяет** этап 1, а дополняет
его как ускоритель + reconciliation остаётся на poll-diff.

## 7. Открытые вопросы (нужно решить до реализации)

1. **Права на чужие приватные репо.** Poll-diff всё ещё требует, чтобы токен бота имел доступ к репозиторию (для
   приватных — нет). Ограничение уже существует сейчас, этим ТЗ не решается.
2. **Объём БД.** SQLite тянет постоянный рост таблицы `commits` без проблем до сотен тысяч строк, но при множестве
   анализируемых профилей стоит заранее прикинуть — возможно, имеет смысл TTL-очистка коммитов старше N лет (`period` и
   так ограничен 730 днями в `analyze.py::MAX_ANALYSIS_DAYS`, можно опираться на него).
3. **Формат `/analyze` с репозиторием** — команда или только через меню? Команда удобнее для степенных юзеров, меню —
   для остальных. Можно сделать оба.
4. **Нужен ли OAuth (этап 2) вообще на первой итерации?** Он ощутимо увеличивает scope (флоу авторизации, хранение чужих
   токенов, безопасность). Предлагаю явно вынести в отдельное ТЗ и не блокировать этап 1.
5. **`needs_full_resync` UX** — если он взводится (force-push), пользователь должен понимать, почему следующий отчёт по
   этому репо "не мгновенный".

## 7.1. Решения по итогам обсуждения (зафиксировано)

- **Движок сбора не трогаем.** `GitHubService`/`GitHubClient`/`TokenPool` остаются как есть — async, уже умеют
  concurrency через `asyncio.Semaphore`. Новый код (`src/core/sync.py`, `src/core/report.py`) — тонкий слой поверх них,
  не переизобретает сбор.
- **Дозагрузка идёт через тот же worker pool.** И poll-diff (обычный `/analyze`), и обработка вебхука (
  `handle_push_event`) — обе задачи в `WorkerSettings.functions`, один arq-пул, один процесс `worker`. Отдельного
  движка/процесса под вебхуки нет — `src/webhook/server.py` только валидирует подпись и кладёт payload в очередь.
- **Бэкфилл всегда полный, не по периоду запроса.** При первой синхронизации репозитория тянется вся история в пределах
  `settings.max_backfill_days` (константа, по умолчанию 730 дней = текущий `MAX_ANALYSIS_DAYS`), а не только то, что
  попросили в конкретном `/analyze`. Отчёт по конкретному периоду/репозиторию — потом просто фильтр по уже собранным
  данным. Реализовано в `worker/tasks.py::analyze_github_user` (`_backfill_start_iso()`).
- **Вебхуки — два независимых профиля использования:**
    - **Для владельца бота ("для меня")** — рабочий путь уже есть: `src/webhook/register.py`, разовый скрипт,
      регистрирует хук на СВОИ репозитории (`OWNER_GITHUB_USERNAME`) через `GITHUB_TOKEN` с правами `admin:repo_hook`.
      Никакого OAuth не требуется — токен и так твой.
    - **Для остальных пользователей бота** — по умолчанию выключено (`WEBHOOK_ENABLED=false`), работает только
      poll-diff. В README описать как опциональную продвинутую фичу: пользователь, который хочет вебхуки на СВОИ
      репозитории, может сам прогнать `register.py` со своим токеном (положить его в свой `.env`/инстанс бота) — то есть
      де-факто self-hosting-way, а не встроенный OAuth-флоу в самом боте. Полноценный OAuth (чтобы это работало "из
      коробки" в мультитенантном сценарии) — вынесен в отдельное ТЗ, не блокирует этот этап.
- **Приватные репозитории чужих профилей** — не в скоупе, ограничение остаётся как и было (токен бота видит только то,
  что видит).
- **Рост БД** — не оптимизируем заранее, чистка/партиционирование — когда реально понадобится.

## 8. Порядок реализации (предлагаемый)

1. Таблицы `tracked_repos` + `commits`, миграция/удаление мёртвой `commit_cache`.
2. `sync_repo_commits()` + `build_analysis_result()` — ядро этапа 1.
3. Переключить `analyze_github_user` на новый пайплайн (sync → read from DB), убрать fingerprint-логику всего профиля.
4. UI: выбор "все репозитории / один репозиторий" в меню + `/repos`.
5. "Быстрый путь" без ARQ для свежих данных.
6. (Опционально, отдельным проектом) OAuth + вебхуки — этап 2.
