# ADR-001: Переход с PyGithub на собственный GraphQL-движок

## Статус

Принято. Реализовано в ветке `feature/github-engine`, смёржено в `main`.

## Контекст

Старый collector (`src/core/collector.py`, до миграции) использовал
PyGithub поверх REST API: `ThreadPoolExecutor` для параллелизма,
`get_commits(author=...)` для истории, обязательный REST-запрос за
файлами для каждого коммита без разбора. Это работало, но:

- REST не умеет сообщать реальную стоимость запроса заранее — только
  постфактум через заголовки;
- один коммит = один REST-запрос за файлами, даже если это чистое
  переименование без изменения содержимого;
- ротация токенов была round-robin вслепую, без знания реального
  remaining/reset_at на момент запроса.

## Решение

Собственный async-слой (`src/github/`) поверх GraphQL как основного
источника данных, с REST только для деталей файлов там, где это
оправдано (`needs_rest_details()`, см. `filters.py`):

- `client.py` — единая точка HTTP (httpx + HTTP/2);
- `auth.py` — TokenPool с раздельным учётом REST/GraphQL лимитов;
- `ratelimit.py` — pre-flight оценка стоимости GraphQL-запроса;
- `models.py` — Repository / CommitHeader (лёгкий) / CommitDetails (тяжёлый);
- `service.py` — единственная точка входа для остального проекта.

## Подтверждение эквивалентности

`scripts/compare_engines.py` (удалён после завершения миграции) прогонялся
на `mageri9/2024-01-01`: 365/365 коммитов, 11/11 репозиториев, полное
совпадение после исправления двух багов, найденных в процессе сравнения:

1. GraphQL `history()` без фильтра по автору считал коммиты всех
   контрибьюторов репозитория, не только анализируемого пользователя —
   добавлен `author: { id: $authorId }` в `REPOSITORY_COMMIT_HISTORY`.
2. Repository-level Redis-кеш (Квест 3.3) не учитывал `author_id` в
   ключе — потенциальная утечка чужой (отфильтрованной под другого
   пользователя) истории между разными анализами одного репозитория.

Осталось одно **осознанно принятое** расхождение: коммиты с
`additions == deletions == 0` (чистые переименования, chmod, создание
пустых файлов) не добираются за списком файлов через REST — экономия
запросов признана важнее полноты данных для этого редкого случая
(см. `needs_rest_details()` в `filters.py`).

## Последствия

Удалено: `src/core/collector.py` (старая версия), `token_rotator.py`,
`fingerprint.py`, `PyGithub` из зависимостей.

Добавлено: `src/github/fingerprint.py` — та же дедупликация по
`SHA256(repo:pushed_at)`, но через `GitHubService` вместо отдельного
PyGithub-соединения.

Отложено (не в скоупе этой миграции): `RateLimiter` из `ratelimit.py`
пока не подключён как admission control перед стартом партии запросов;
GraphQL aliases batching для объединения нескольких репозиториев в один
HTTP-запрос.
